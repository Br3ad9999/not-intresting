--------------------------------------------------
1. PROJECT OVERVIEW
--------------------------------------------------

This is a Gym Management System built using:

- Frontend: HTML, CSS, JavaScript
- Backend: Node.js with Express
- Database: PostgreSQL
- Authentication style: session-based login using express-session

Main project files:

- index.html
  Login and signup page

- index.css
  Styles for index.html

- dashboard.html
  Main dashboard page after login

- dashboard.css
  Styles for dashboard.html

- server.js
  Main backend server with all API routes

- db.js
  PostgreSQL connection setup and transaction helper

- sql/schema.sql
  All CREATE TABLE statements

- sql/seed.sql
  Demo/sample data

- sql/project_queries.sql
  Important SQL queries for display, update, delete, and GROUP BY + ORDER BY report

- .env
  Local environment variables such as PostgreSQL username, password, database, and port


--------------------------------------------------
2. HIGH LEVEL FLOW
--------------------------------------------------

The project works like this:

1. User opens the website.
2. The server sends index.html.
3. The browser loads index.css and JavaScript inside index.html.
4. If the user signs up or logs in, JavaScript sends a fetch request to backend API routes in server.js.
5. The backend checks PostgreSQL.
6. If login/signup is successful, the backend stores session data.
7. The browser moves to dashboard.html.
8. dashboard.html calls /api/auth/me to learn who is logged in.
9. If the logged-in user is a normal member, they see member information only.
10. If the logged-in user is an admin, they also see an admin panel.
11. The admin panel can:
   - display protected database data
   - update member records
   - delete contact messages
   - show a GROUP BY + ORDER BY report

4.1 db.js
----------

This file handles PostgreSQL connection setup.

What it does:

- imports Pool from pg
- loads environment variables from .env using dotenv
- checks whether SSL should be used
- builds database config using either:
  - DATABASE_URL
  OR
  - individual PGHOST, PGPORT, PGUSER, PGPASSWORD, PGDATABASE values

Important pieces:

- pool
  Connection pool for PostgreSQL

- query(text, params)
  Helper for normal SQL queries

- withTransaction(work)
  Helper for multi-step operations
  It:
  - opens a DB client
  - starts BEGIN
  - runs the provided work
  - COMMITs on success
  - ROLLBACKs on error

Why transactions matter:

Signup inserts into multiple tables:

- users
- members
- memberships

If one insert fails, transaction rollback keeps data consistent.

-----------------------

4.4 Utility functions in server.js
----------------------------------

- asyncHandler(handler)
  Wraps async route handlers so errors go to Express error handler.

- requireAuth(req, res, next)
  Blocks routes if no user session exists.

- requireAdmin(req, res, next)
  Blocks routes unless session exists and role is admin.

- normalizeName(value)
  Cleans extra spaces from names.

- buildBaseUsername(fullName, email)
  Creates a username base from email or name.

- generateUniqueUsername(client, fullName, email)
  Keeps checking database until it finds a unique username.

- fetchCurrentUserProfile(userId, runner)
  Reads combined user/member/membership/branch data using joins.
  This is important because frontend needs a single user object with:
  - username
  - email
  - role
  - member details
  - branch details
  - latest membership details

- serializeUser(profile)
  Converts raw SQL row into cleaner JSON for frontend.

5.2 GET /api/plans
-------------------

Purpose:
- send active membership plans to signup page

Database query:
- reads plan_id, plan_name, monthly_fee, duration_months, description
  from membership_plans
  where is_active = TRUE
  ordered by monthly_fee, plan_name

Frontend usage:
- index.html loadPlans()


5.3 POST /api/auth/signup
--------------------------

Purpose:
- create a new member account

Input from frontend:
- fullName
- email
- password
- planId

Checks:
- all fields must be present
- password length must be at least 6
- email must not already exist
- plan must exist and be active

Transaction steps:

1. Check duplicate email in users
2. Check selected plan in membership_plans
3. Generate unique username
4. Hash password using bcrypt
5. Pick first branch as default branch
6. Insert into users
7. Insert into members
8. Insert into memberships
9. Read the full profile back using fetchCurrentUserProfile

Session effect:
- server stores:
  - userId
  - role

Frontend usage:
- index.html handleSignup()


5.4 POST /api/auth/login
-------------------------

Purpose:
- log in an existing user

Input from frontend:
- identifier
  can be username or email
- password

Checks:

1. find matching user by username or email
2. compare given password with password_hash using bcrypt.compare()
3. if correct:
   - create session
   - fetch full profile
   - send it back

Frontend usage:
- index.html handleLogin()
- index.html guestLogin()


5.5 POST /api/auth/logout
--------------------------

Purpose:
- destroy session and log out user

What happens:
- req.session.destroy()
- clearCookie('connect.sid')

Frontend usage:
- dashboard.html logout()

6. HOW FRONTEND AND BACKEND COMMUNICATE
--------------------------------------------------

Communication method:

- JavaScript fetch() calls
- JSON request bodies
- JSON responses
- HTTP methods:
  - GET
  - POST
  - PUT
  - DELETE

Examples:

1. Signup communication

Frontend:
- handleSignup() sends POST /api/auth/signup with:
  {
    fullName,
    email,
    password,
    planId
  }

Backend:
- validates input
- inserts into database
- creates session
- returns:
  {
    message: "...",
    user: { ... }
  }


2. Login communication

Frontend:
- handleLogin() sends POST /api/auth/login with:
  {
    identifier,
    password
  }

Backend:
- checks users table
- verifies bcrypt hash
- stores session
- returns profile JSON


3. Session communication

After login, browser automatically keeps session cookie.

Important idea:
- frontend does NOT manually store password
- backend session cookie identifies the logged-in user

Then later requests like /api/auth/me or /api/admin/overview can identify the user from session.


4. Admin communication

Frontend:
- dashboard checks role returned by /api/auth/me
- if role is admin, it shows admin button and loads admin data

Backend:
- requireAdmin middleware blocks normal users

9. IMPORTANT SQL QUERIES USED IN THE PROJECT
--------------------------------------------------

All submission-related queries are in sql/project_queries.sql.


9.1 Display query
------------------

Purpose:
- display members with branch and current plan

Query:

SELECT
  m.member_id,
  m.full_name,
  u.username,
  u.email,
  b.branch_name,
  current_membership.plan_name,
  current_membership.status,
  m.join_date
FROM members m
JOIN users u ON u.user_id = m.user_id
LEFT JOIN branches b ON b.branch_id = m.branch_id
LEFT JOIN LATERAL (
  SELECT mp.plan_name, ms.status
  FROM memberships ms
  JOIN membership_plans mp ON mp.plan_id = ms.plan_id
  WHERE ms.member_id = m.member_id
  ORDER BY (ms.status = 'active') DESC, ms.end_date DESC NULLS LAST, ms.membership_id DESC
  LIMIT 1
) AS current_membership ON true
ORDER BY m.join_date DESC, m.member_id DESC;

Explanation:

- starts from members
- joins users for login/email info
- joins branches for branch info
- uses LATERAL join to choose the latest or active membership row per member
- orders newest members first


9.2 GROUP BY + ORDER BY query
------------------------------

Purpose:
- satisfy the DBMS requirement for grouped display data
- show report by plan

Query:

SELECT
  mp.plan_name,
  COUNT(DISTINCT ms.membership_id) FILTER (WHERE ms.status = 'active') AS active_members,
  COALESCE(SUM(p.amount) FILTER (WHERE p.payment_status = 'paid'), 0) AS collected_amount
FROM membership_plans mp
LEFT JOIN memberships ms ON ms.plan_id = mp.plan_id
LEFT JOIN payments p ON p.membership_id = ms.membership_id
GROUP BY mp.plan_id, mp.plan_name, mp.monthly_fee
ORDER BY active_members DESC, mp.monthly_fee ASC;

Explanation:

- groups rows by membership plan
- counts active memberships in each plan
- sums paid payment amount
- orders plans by:
  1. highest active member count
  2. then lower monthly fee

This is shown in admin plan report on dashboard.


9.3 Update query
-----------------

Purpose:
- update a member's membership

Query:

UPDATE memberships
SET plan_id = 3, status = 'active', end_date = CURRENT_DATE + INTERVAL '30 days'
WHERE member_id = 2 AND membership_id = (
  SELECT membership_id
  FROM memberships
  WHERE member_id = 2
  ORDER BY membership_id DESC
  LIMIT 1
);

Explanation:

- finds latest membership row for member 2
- changes plan and status
- extends end date

In the live app, backend route PUT /api/admin/members/:memberId performs similar logic dynamically.


9.4 Delete query
-----------------

Purpose:
- delete one contact message

Query:

DELETE FROM contact_messages
WHERE message_id = 2;

Explanation:

- deletes a specific message row by id

In the live app, backend route DELETE /api/admin/messages/:messageId performs this.

11. HOW AUTHENTICATION WORKS
--------------------------------------------------

Passwords:

- user enters plain password in frontend
- backend hashes on signup using bcrypt.hash()
- database stores only password_hash

Login:

- user enters username/email + password
- backend finds stored hash
- bcrypt.compare() checks match

Session:

- if login succeeds, backend stores:
  - userId
  - role
  in session

- browser keeps session cookie
- future requests automatically carry session cookie

Role handling:

- member role:
  normal dashboard

- admin role:
  normal dashboard + admin control panel
